# RSVP API — бэкенд анкеты

Легковесный Node.js-сервис (без npm-зависимостей) в Docker-контейнере. Проверяет
секретные слова гостей и записывает ответы анкеты в Google-таблицу. Наружу не
торчит: доступен только через nginx сайта (`location /api/` → `http://<контейнер>:3000`).

## Эндпоинты

| Метод | Путь               | Тело                      | Ответ                                 |
|-------|--------------------|---------------------------|---------------------------------------|
| POST  | `/api/rsvp/auth`   | `{word}`                  | `{ok, names[], answers, locked}`      |
| POST  | `/api/rsvp/submit` | `{secretWord, …ответы}`   | `{ok}` / `{ok, queued}`               |
| POST  | `/api/rsvp/photo`  | `?word=&g=0\|1` + сырые байты картинки | `{ok, url}` / `{ok:false, error}` |
| POST  | `/api/rsvp/photo/clear` | `{word, g}`          | `{ok}`                                |
| GET   | `/api/rsvp/photo/<file>.png` | —               | сам PNG (прямая ссылка на скачивание) |
| GET   | `/api/health`      | —                         | `{ok, demo}`                          |

**Фото гостей** (`/photo`, `src/photos.js`): загрузка и очистка гейтятся секретным
словом (`findGuest`) — принимаем только от приглашённых; `g` — индекс гостя (0/1,
1 только у пары). Тело — сырые байты **любого** формата (JPEG/PNG/WebP/HEIC…):
libvips (`vipsthumbnail` из пакетов `vips-tools`+`vips-heif`, ставятся в `Dockerfile`)
конвертирует в качественный PNG с даунскейлом до 1600px и авто-разворотом по EXIF.
Файлы лежат в `/data/photos` (docker volume), раздаются по неугадываемому имени.
Ссылка на фото пишется в колонки `K/L` «Ответов» формулой `=IMAGE("url")` (миниатюра
в ячейке + url в строке формул); при сбое таблицы — фото-outbox с ретраями. Замена
перезаписывает файл и ячейку (работает и после «Заполнено»). Прямые ссылки собираются
из `PUBLIC_BASE_URL` (по умолчанию `https://wedding.simitka.io`).

`/auth` возвращает не только имена, но и сохранённые ответы гостя (`answers` —
разобранная строка таблицы или `null`) и флаг замка (`locked` = «Заполнено?»):
по ним фронтенд решает экран (замок → read-only, всё заполнено → чек, иначе →
незаполненный шаг).

Полная форма payload сабмита — см. `rsvpBuildPayload()` в `index.html`; сервер
всё равно пересобирает запись сам: имена берёт из списка приглашённых (первые
три колонки «Ответов»), значения — из белых списков (`sanitizePayload` в
`src/rsvp.js`).

## Как хранятся данные

Google-таблица — две вкладки. Отдельного списка «Гости» больше нет: список
приглашённых живёт в первых трёх колонках «Ответы на rsvp» (при пропаже вкладки
бэкенд создаст пустую с заголовками; «Сводку» — нет).

- **«Ответы на rsvp»** — единственная вкладка-источник. Колонки
  `A Секретное слово | B Гость 1 | C Гость 2` — **список приглашённых, ведётся
  руками** (одно имя = личное приглашение, два = пара; слова сравниваются без
  регистра, «ё»→«е», только буквы/цифры; кеш списка — 60 секунд; строка без
  имени в auth не пройдёт). Остальные колонки дописывает бэкенд при отправке
  анкеты, обновляя ту же строку:
  `D Приедут? | E Трансфер | F Мест для попутчиков | G Ночёвка | H Оплата домика |
  I Заполнено? | J Обновлено`. Значения совпадают с выпадающими списками таблицы:
  «Приедут?» = `Нет / Да, Гость 1 / Да, Гость 2 / Да, оба`; «Трансфер» =
  `Нужен / Не нужен`; «Ночёвка» = `Остаюсь / Не остаюсь`; «Оплата домика» =
  `Заплачу / Не заплачу` (пусто, если не остаётся); «Мест для попутчиков» — число
  0–7 (пусто, если нужен трансфер). «Заполнено?» — чекбокс: отправка анкеты ставит
  `TRUE` (замок — на сайте гость видит read-only); снимите галочку, чтобы он снова
  мог редактировать. Append идёт с `insertDataOption=OVERWRITE` — вставка строк
  сдвинула бы ссылки формул сводки.
- **«Сводка»** — формулы поверх «Ответов»: приглашено/ответили/ещё не ответили,
  сколько людей едет, на трансфер, свободных мест у попутчиков, ночует, кто платит
  за домик. Людей считает из «Приедут?» (`Да, оба` = 2 человека). Руками не
  заполняется — перестраивается скриптом `scripts/setup-sheets.mjs` (он же
  переносит старую вкладку «Гости» в «Ответы» и удаляет её).
- **Локальный журнал** (`/data/answers.log.jsonl`) — каждый сабмит дублируется
  на диск контейнера (страховка). Если таблица недоступна, запись попадает в
  `/data/outbox.json` и досылается ретраем каждые 30 секунд.

## Переменные окружения

| Переменная               | По умолчанию                                   | Зачем                              |
|--------------------------|------------------------------------------------|------------------------------------|
| `GOOGLE_SA_JSON_BASE64`  | —                                              | ключ сервис-аккаунта (base64 JSON) |
| `GOOGLE_SA_JSON_PATH`    | —                                              | либо путь к смонтированному JSON   |
| `SPREADSHEET_ID`         | `13UFjGy42iBV9rhGtWehsmoCnbq1rJSWOBPR_mCXTX0U` | таблица с ответами                 |
| `ANSWERS_SHEET`          | `Ответы на rsvp`                               | вкладка-источник (гости + ответы)  |
| `PUBLIC_BASE_URL`        | `https://wedding.simitka.io`                   | база прямых ссылок на фото (`=IMAGE`) |
| `PORT` / `DATA_DIR`      | `3000` / `/data`                               |                                    |

Без ключа сервис-аккаунта API работает в **демо-режиме**: пароли
«хинкали»/«сациви», ответы пишутся только в журнал.

## Доступ к таблице (что нужно сделать один раз)

Таблица принадлежит `nikita2000nikita@gmail.com`. Чтобы сервис писал в неё:

1. Под этим аккаунтом зайти на <https://console.cloud.google.com> → создать
   проект (например, `wedding-rsvp`).
2. **APIs & Services → Library** → включить **Google Sheets API**.
3. **APIs & Services → Credentials → Create credentials → Service account**
   (имя любое, роли не нужны) → в созданном аккаунте вкладка **Keys →
   Add key → JSON** — скачается файл ключа.
4. В самой таблице нажать **Настройки доступа** и добавить e-mail сервис-аккаунта
   (`...@...iam.gserviceaccount.com`) как **Редактора**.
5. Ключ положить в Keychain и не хранить в plaintext:

   ```bash
   security add-generic-password -U -s wedding-rsvp-gsa -a token -w "$(base64 -i ~/Downloads/wedding-rsvp-*.json)"
   ```

   Дальше значение `GOOGLE_SA_JSON_BASE64` достаётся так:

   ```bash
   security find-generic-password -s wedding-rsvp-gsa -a token -w
   ```

## Локальный запуск

```bash
docker compose up --build          # http://localhost:8080 (сайт + /api)
curl -s localhost:8080/api/health  # {"ok":true,"demo":true}
```

## Прод (Dokploy)

Второе приложение в том же проекте `wedding`, источник — тот же репозиторий,
build type = Dockerfile с путём `backend/Dockerfile` и контекстом `backend`.
Публичный домен ему **не нужен** — сайт ходит к нему по имени сервиса внутри
docker-сети (env `RSVP_API_HOST` у приложения сайта = appName бэкенда).
Env бэкенда: `GOOGLE_SA_JSON_BASE64` (+ при желании volume на `/data`).
